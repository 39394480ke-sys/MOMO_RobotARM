# 开发板硬件恢复手册

本文记录 Ubuntu ARM64 开发板作为 MOMO 机械臂上位机时的恢复步骤。默认不执行真实动作，先恢复串口、驱动、环境和只读扫描。

## 当前标准状态

- 系统 Python 不跑项目，项目使用 `momo_rebot` / Python 3.11。
- Web 默认模式保持 `dry_run`。
- 真实舵机后端默认使用轻量 SDK：`ARM_SERVO_BACKEND=sdk`。
- 开发板真实串口统一使用稳定别名：`/dev/momo-servo`。
- 控制板 USB 设备为 `1a86:55d3 QinHeng USB Single Serial`。
- 正确驱动应为 `usb_ch343`，不是 `option`。

`.env` 推荐值：

```bash
ARM_DEFAULT_MODE=dry_run
ARM_REAL_DRY_RUN=true
ARM_SERVO_BACKEND=sdk
ARM_ROBOT_PORT=/dev/momo-servo
```

## 一键恢复 CH343 串口

在开发板仓库根目录执行：

```bash
bash scripts/setup_qinheng_55d3_serial.sh
```

该脚本会：

- 安装编译依赖 `build-essential git flex bison`。
- 编译 WCH 官方 `ch343.ko`。
- 安装模块到 `/lib/modules/$(uname -r)/extra/ch343.ko`。
- 写入 `/etc/modules-load.d/ch343.conf`。
- 写入 `/etc/udev/rules.d/98-momo-servo-ch343.rules`。
- 禁用旧的 `qinheng-55d3-serial-bind.service`。
- 创建稳定串口 `/dev/momo-servo`。

如果插着控制板时仍显示 `Driver=option`，先拔插控制板 USB，再检查。

## 驱动与串口检查

```bash
lsusb | grep -i '1a86:55d3'
lsusb -t | grep -E 'usb_ch343|option'
ls -l /dev/momo-servo /dev/ttyCH343USB*
```

成功状态应类似：

```text
Driver=usb_ch343
/dev/momo-servo -> ttyCH343USB0
```

如果看到 `Driver=option`，说明控制板仍被调制解调器驱动占用，需要重新运行恢复脚本并拔插 USB。

## 只读舵机扫描

只读扫描不会写 `Goal_Position`，也不会移动舵机：

```bash
source "$HOME/miniforge3/etc/profile.d/conda.sh"
conda activate momo_rebot
python 真实舵机控制/诊断舵机总线_lightweight_sdk.py --port /dev/momo-servo --no-gripper
```

成功标准：

```text
ping 结果：{10: 777, 11: 777, 12: 777, 13: 777, 14: 777, 15: 777}
```

连续三次扫描结果一致后，才允许进入真实模式读状态。

## Web 真实硬件检查

启动 Web 后打开：

```text
http://开发板IP:8010/web/
```

进入“设置”，点击“真实硬件检查”。检查项包括：

- `/dev/momo-servo` 是否存在。
- CH343 是否接管。
- real 模式依赖是否就绪。
- 标定文件是否存在。
- ID `10-15` 是否可只读扫描。
- 当前 `Present_Position` raw 读数。

## 真实动作安全顺序

真实动作只按以下顺序推进：

1. Web 保持 dry-run。
2. 真实硬件检查通过。
3. real connect 只读状态通过。
4. 单关节微动，首轮 `0.5°`。
5. 每次动作前后记录 `raw_present_position`。
6. 测试后恢复 dry-run。

禁止在未确认前执行：

- 多关节动作。
- Home / 回零。
- 动作库回放。
- 视觉真实跟随。
- 直接写 raw 舵机寄存器。
